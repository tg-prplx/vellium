# Vellium Web Service: E2EE, BYOK, Regulation, and Implementation Plan

> Status: future design note, not current application behavior.  
> Last reviewed: 2026-07-19.  
> Jurisdiction assumed by this note: an operator established in Germany and a
> service offered to users in the EU/EEA.

This document records the proposed path for turning Vellium into a private web
service without turning it into a conventional provider-controlled AI SaaS. It
also records the legal and personal-exposure trade-offs that should be resolved
before implementation starts.

This is an engineering and product plan, not a legal opinion. German counsel
should review the final product, website, and operating entity before a public
commercial launch.

## 1. Recommended product model

The recommended web edition is a private, general-purpose character writing and
roleplay workbench with these properties:

- an account is created from a high-entropy key generated in the browser;
- no email address, username, or reusable password is required;
- the server stores an account identifier and authentication verifier, not the
  master key;
- settings, chats, characters, LoreBooks, provider credentials, and attachments
  are encrypted in the browser before upload;
- the server stores only encrypted chunks, revisions, quotas, subscription state,
  and the minimum operational metadata;
- users bring and pay for their own model providers (BYOK);
- direct browser-to-provider generation is preferred where provider CORS permits;
- an optional relay may exist for compatibility, but it is a clearly disclosed
  privacy downgrade because the relay sees plaintext in memory;
- there is no public chat sharing, character marketplace, feed, or public NSFW
  catalog;
- local process execution, arbitrary MCP commands, workspace shell tools, and
  untrusted server plugins are not available in the hosted edition.

The desktop/local-first application remains a separate product surface. The web
edition must not silently weaken the desktop product or migrate existing local
data into the cloud.

## 2. Why the current server cannot simply be exposed

Vellium already has a useful transport foundation: `npm run headless` builds the
renderer and server bundle, then serves the frontend and `/api` without Electron.
That makes a web edition practical, but the current persistence and account model
are intentionally single-user:

- `server/routes/account.ts` verifies the most recently created local account and
  does not establish isolated multi-user sessions;
- `settings` has a fixed `id = 1` singleton;
- providers, chats, characters, LoreBooks, writer data, and files are not scoped
  by an account or tenant;
- `providers.api_key_cipher` currently receives the submitted API key directly;
  its name does not mean that encryption is currently performed;
- public headless mode uses one deployment-wide Basic Auth credential;
- local MCP, Agents, plugins, filesystem tools, and provider URLs assume a trusted
  local operator rather than hostile anonymous internet users.

The existing public/headless flags are suitable for a single trusted owner behind
TLS. They are not a multi-user cloud security boundary.

## 3. Client-side key design

### 3.1 Master key

The browser generates a uniformly random 256-bit `MASTER_KEY` using Web Crypto.
It is displayed once as a recovery string or word sequence and never leaves the
client.

The master key must not be:

- generated by the server;
- written to application logs, analytics, URLs, crash reports, or clipboard
  telemetry;
- stored in plaintext server-side;
- sent to the server for login;
- used directly for both authentication and encryption.

Losing the master key means losing the ability to decrypt the workspace. The UI
must state this before account creation and offer an encrypted recovery-file
download.

### 3.2 Domain-separated derived keys

Derive independent keys with HKDF-SHA-256 and versioned context labels:

```text
MASTER_KEY (browser only)
  |
  +-- HKDF(info = "vellium-auth-v1") --> AUTH_SECRET
  |
  +-- HKDF(info = "vellium-data-kek-v1") --> DATA_KEK
```

The server may receive `AUTH_SECRET` over TLS during a basic login flow and store
only a slow or peppered verifier. Possession of `AUTH_SECRET` must not allow the
server to derive `DATA_KEK`. A reviewed PAKE such as OPAQUE can replace this basic
flow later if the threat model requires the server never to receive a reusable
authentication secret.

Do not send `hash(MASTER_KEY)` as the login credential while also storing that
same value server-side. In that design, a database leak turns the stored hash into
a replayable password.

### 3.3 Envelope encryption

Generate a random data-encryption key (`DEK`) for the workspace. Use `DEK` to
encrypt records, and wrap `DEK` with `DATA_KEK`:

```text
DATA_KEK (derived in browser)
  |
  +-- encrypts/wraps --> workspace DEK
                          |
                          +-- encrypts --> settings chunk
                          +-- encrypts --> chat chunks
                          +-- encrypts --> characters chunk
                          +-- encrypts --> LoreBook chunks
                          +-- encrypts --> attachment chunks
```

This permits account-key rotation by rewrapping the DEK instead of re-encrypting
the entire workspace.

Use an authenticated encryption mode supported by Web Crypto, initially
AES-256-GCM. Every encryption operation requires a unique random nonce. Bind the
following values as authenticated additional data (AAD):

- crypto format version;
- account/workspace identifier;
- chunk identifier and type;
- revision number;
- algorithm identifier.

Decryption must fail closed when authentication fails. Never attempt to salvage
or parse unauthenticated plaintext.

### 3.4 Chunking and revisions

A single monolithic blob is acceptable only for a prototype. Production storage
should use independently encrypted chunks because a monolith causes full-workspace
uploads, poor conflict handling, and large corruption domains.

Suggested chunk classes:

- workspace manifest and preferences;
- provider profiles and credentials;
- chat index;
- one chunk or bounded chunk set per chat/branch;
- character index and character records;
- LoreBooks and knowledge metadata;
- writer projects;
- encrypted attachment objects.

Every write uses optimistic concurrency (`If-Match`, ETag, or expected revision).
The client rejects server rollback to an older signed/authenticated revision. Two
devices must produce a visible merge conflict instead of silently overwriting one
another.

## 4. Authentication and account lifecycle

### Registration

1. Generate `MASTER_KEY`, `AUTH_SECRET`, and `DATA_KEK` in the browser.
2. Generate a non-secret random account identifier.
3. Generate and encrypt the initial workspace DEK.
4. Send the account identifier, authentication material required by the selected
   protocol, wrapped DEK, and encrypted initial manifest.
5. Show the recovery key once and require an explicit acknowledgement.

### Login

1. The user enters the master/recovery key locally.
2. The browser derives `AUTH_SECRET` and `DATA_KEK`.
3. The browser authenticates without sending `MASTER_KEY` or `DATA_KEK`.
4. The server creates a short-lived session using a `Secure`, `HttpOnly`,
   `SameSite` cookie.
5. The browser downloads and decrypts the encrypted manifest.

### Rotation and recovery

- Authentication-session rotation must not require data re-encryption.
- Master-key rotation rewraps the workspace DEK and invalidates all sessions.
- There is no operator recovery of a lost encryption key.
- An optional recovery code is another client-held wrapping key, not a server
  backdoor.
- Account deletion removes ciphertext, wrapped keys, auth verifiers, sessions,
  billing linkage where legally possible, and queued deletion backups according
  to a documented retention schedule.

## 5. Generation and provider privacy modes

### Direct mode (preferred)

```text
Browser --prompt + user API key--> Provider
Browser --encrypted result--------> Vellium storage
```

Vellium does not receive the provider credential, prompt, or generated response.
This is the strongest privacy mode but requires provider support for browser CORS
and safe browser-issued credentials.

### Relay mode (compatibility fallback)

```text
Browser --prompt + user API key--> Stateless Vellium relay --> Provider
Browser <--stream----------------- Stateless Vellium relay <-- Provider
```

Relay mode must be opt-in and accurately described:

- transport is protected with TLS;
- provider credentials and content exist in relay memory during the request;
- request and response bodies are never persisted or logged;
- error tracking receives structured error codes, not request bodies or headers;
- aggressive time, byte, concurrency, and destination limits apply;
- SSRF defenses re-resolve and validate destinations and redirects;
- localhost, private, link-local, metadata, and non-HTTP(S) destinations are
  blocked;
- the relay cannot honestly be marketed as zero knowledge or end-to-end encrypted.

Local endpoints such as LM Studio or KoboldCpp on a user's computer are not
reachable from a cloud server. They require direct browser CORS support or a
separately installed, authenticated local connector using an outbound tunnel.

## 6. Hosted-edition capability policy

| Capability | Hosted web policy |
| --- | --- |
| Chat, branches, RP, multi-character | Supported client-side |
| Characters, personas, LoreBooks | Supported as encrypted data |
| Writer projects | Supported as encrypted data |
| Chat export/import | Performed client-side after decryption |
| Provider profiles and API keys | Encrypted client-side |
| Direct provider generation | Preferred when CORS permits |
| Provider relay | Optional, stateless, explicitly disclosed |
| RAG | Local/client-side first; server cannot index ciphertext |
| Background generation | Not available without an open client or trusted relay |
| Server-side search | Not available over encrypted content |
| Arbitrary MCP stdio commands | Disabled |
| Shell, git, workspace, and Agents tools | Disabled |
| User-installed server plugins | Disabled |
| Curated remote MCP over HTTPS | Future, allowlisted and separately reviewed |
| Public chat/character sharing | Out of scope for the private edition |
| Electron pet/window/native features | Desktop-only |

Feature gates must be enforced by the server as well as hidden in the renderer.
Do not depend on a client-side flag to protect command execution or internal
network access.

## 7. Browser-client trust limitation

Client-side encryption protects the database and normal server processing, but a
web server controls the JavaScript it delivers. A malicious or compromised future
frontend build could capture the master key before encryption.

Therefore public claims must be limited to what the architecture proves. Do not
claim that the operator can never access data under every threat model.

Risk-reduction measures:

- publish the client source and crypto format;
- keep cryptographic code small and independently reviewed;
- publish test vectors and reproducible build instructions;
- use strict CSP and remove arbitrary third-party scripts from the key-handling
  origin;
- isolate billing, analytics, support, and documentation on origins that cannot
  execute code in the encrypted workspace;
- consider a signed desktop client or browser extension for users requiring a
  stronger malicious-server threat model;
- version the crypto format and provide an offline decrypt/export utility.

## 8. Data the operator still processes

E2EE does not make the operator legally or technically invisible. The service may
still process:

- account identifier and authentication verifier;
- session identifiers;
- IP address and bounded security logs;
- ciphertext size, revision, and timestamps;
- subscription status and payment-provider references;
- support messages voluntarily sent by the user;
- plaintext provider traffic in relay mode;
- abuse notices and lawful authority requests.

Do not describe an account as fully anonymous when payments, support, or security
logs can link it to a person. "Account without a personal profile" or
"pseudonymous account" is more accurate.

## 9. German and EU regulatory posture

### 9.1 Impressum and operator exposure

For a commercial digital service operated from Germany, § 5 DDG requires an
easily recognisable, directly accessible, permanently available legal notice. It
includes the operator's name and a physical address at which the business is
established, plus direct electronic contact information and applicable company or
register details.

This is a central product risk for a solo developer:

- an individual business generally exposes the proprietor's real name and a
  serviceable street address;
- a post-office box is insufficient;
- a nominal virtual mailbox is not automatically a valid business establishment;
- a genuine office/coworking business address may protect the home address only
  if it is legally usable as the actual establishment and reliably accepts legal
  service;
- a UG/GmbH can publish its company address instead of the founder's home address,
  but it still publishes the legal form and authorised representative/manager;
- using an established publisher/operator is viable only if that party genuinely
  operates the service and assumes the related obligations.

The business address and entity structure must be resolved before a public beta,
not after the landing page is indexed.

### 9.2 GDPR

Treat ciphertext conservatively as protected personal data rather than assuming
that encryption removes it from GDPR. The operator is clearly responsible for
account, IP, session, payment, support, and security metadata.

Minimum launch work:

- a precise privacy notice describing direct and relay modes separately;
- documented lawful bases for account/service delivery, security logging, billing,
  and optional analytics;
- data minimisation and explicit retention schedules;
- client-assisted access/export and server-side deletion workflows;
- data-processing agreements with hosting, CDN, payment, email/support, and error
  monitoring vendors;
- an international-transfer assessment where vendors or relay destinations are
  outside the EEA;
- a breach-response process that distinguishes ciphertext-only exposure from
  compromise of authentication, frontend delivery, or key material;
- a DPIA screening, with a full DPIA if the final scale and use cases make high
  risk likely;
- technically necessary cookies only on the encrypted workspace origin; optional
  analytics or marketing must not load before any required consent.

Encryption materially reduces impact and demonstrates privacy by design. It is a
safeguard, not a general regulatory exemption.

### 9.3 DSA

Encrypted storage may still be classified as a hosting component. With no public
sharing, the service is less likely to be an online platform that disseminates
user information to the public, but the final classification is fact-specific.

The service should provide:

- clear terms and a human-reachable contact point;
- an electronic notice channel for specific allegedly illegal material/activity;
- a process for preserving and responding to valid specific authority orders;
- an account-level disable/delete action when individual encrypted content cannot
  technically be inspected or removed;
- clear explanations when an account is restricted.

The DSA does not impose a general duty to monitor stored information or actively
search for illegal activity. E2EE must not be weakened to create general scanning.
The operator can provide only the ciphertext and metadata it actually possesses.

### 9.4 AI Act

If Vellium remains a general-purpose UI for user-selected third-party models, does
not market itself for high-risk decisions, and does not train or place its own GPAI
model on the market, the main expected product obligation is transparency around
AI interaction.

The UI should clearly state that responses are produced by the model selected by
the user and may be inaccurate. The legal roles of Vellium, the model provider,
and the user should be reviewed again if Vellium later hosts models, modifies and
rebrands them as its own, or supplies a bundled model service.

### 9.5 German youth protection and adult positioning

The private encrypted tool and an adult-content service are different products.

The recommended private workbench:

- uses neutral product positioning and defaults;
- does not ship pornographic characters, examples, presets, screenshots, or
  landing-page copy;
- lets users configure providers and import private content themselves;
- does not publish or recommend user content;
- does not offer public NSFW discovery or sharing;
- does not claim that operator-side moderation of encrypted chats is possible.

In this model, regulating the contents of private encrypted chats is neither a
technical feature nor the product's compliance strategy. The service controls its
own supplied content, capabilities, marketing, and public surfaces.

If Vellium instead markets itself as adult AI RP, supplies pornographic content,
or provides a public adult catalog, client-side encryption does not remove the
operator's responsibility for the offer it makes. § 4 JMStV permits certain adult
telemedia content only when the provider ensures access solely within a closed
adult user group. A simple "I am 18" checkbox is not a substitute for a qualifying
age-verification system in that adult-service scenario.

Decision: the initial hosted edition is not an adult-content service. A future
adult edition is a separate legal and product track requiring specialist German
review, an accepted age-verification approach, and an operator prepared to assume
that public role.

### 9.6 Abmahnung and formal website risk

German warning-letter risk is not eliminated by E2EE. The common avoidable
surfaces include:

- incomplete or inaccessible Impressum;
- misleading privacy or zero-knowledge claims;
- improperly loaded analytics/cookies;
- missing consumer-contract information;
- unlicensed fonts, images, icons, character art, or copied text;
- trademark-confusing product naming;
- invalid pricing, renewal, cancellation, or withdrawal presentation.

Recent UWG rules limit reimbursement of warning costs in some competitor claims
for digital information duties and small-company GDPR violations, and prohibit
abusive enforcement. They do not eliminate injunctions, defence costs, authority
action, copyright claims, or claims by every other eligible body.

Do not sign a broad cease-and-desist undertaking (`Unterlassungserklärung`) without
independent legal advice. A pre-launch legal review of the actual rendered site is
cheaper than correcting the product under deadline after a notice.

## 10. Operating-entity decision

Before implementation reaches public beta, choose one path explicitly:

1. **German individual operator.** Lowest setup cost, highest personal exposure;
   the proprietor's name and serviceable business address are public.
2. **German UG/GmbH with a genuine business address.** Better separation of the
   home address and liability, but the authorised representative remains public
   and administration costs increase.
3. **Genuine publishing/operating partner.** The partner controls billing and
   operation and assumes provider duties under a real contract; it is not a
   cosmetic Impressum proxy.
4. **No hosted product.** Keep Vellium local/desktop and distribute software rather
   than operate encrypted user storage. A commercial website may still require an
   Impressum, but hosting and content-operation exposure is much smaller.

Do not choose an entity structure primarily to evade a valid legal address or
jurisdiction. It must match who actually controls and operates the service.

## 11. Implementation phases

### Phase 0: product and legal boundary

- Freeze the initial hosted-edition capability matrix.
- Confirm neutral positioning and absence of public user-content surfaces.
- Choose the operating entity and valid business address.
- Obtain short written German advice covering DDG/Impressum, GDPR roles, DSA
  classification, JMStV positioning, consumer terms, and direct versus relay mode.
- Define the exact public privacy claims before designing around them.

Exit criterion: a written go/no-go memo and approved public product description.

### Phase 1: standalone crypto prototype

- Implement browser master-key generation and display.
- Implement HKDF domain separation.
- Implement DEK generation, AES-GCM chunk encryption, and DEK wrapping.
- Publish deterministic test vectors for derivation/wrapping and round-trip tests
  for encrypted chunks.
- Implement offline export/decrypt and corruption detection.
- Perform an independent cryptographic design review before integrating with the
  application.

Exit criterion: the server-side test fixture can store and return blobs but cannot
decrypt any test workspace.

### Phase 2: encrypted storage API

- Add account registration/authentication/session endpoints separate from the
  current local account routes.
- Add manifest/chunk CRUD with expected revisions and bounded payload sizes.
- Add encrypted attachment storage with content-address-independent opaque IDs.
- Add account deletion and backup-retention state.
- Add strict quotas, rate limits, audit events without content, and abuse controls.
- Ensure all object access is scoped by the authenticated account at one server
  boundary.

Exit criterion: cross-account reads/writes, replay, rollback, oversized payloads,
and leaked-verifier login attempts have regression tests.

### Phase 3: renderer web-vault layer

- Add a client vault responsible for all serialization, encryption, and sync.
- Keep decrypted state in memory and minimize browser persistence.
- Migrate chat, character, LoreBook, writer, and settings persistence behind the
  vault interface.
- Add sync status, conflicts, offline changes, retry, and explicit lock/logout.
- Add recovery-key confirmation and unrecoverable-loss UX.

Exit criterion: a fresh browser and a second device can open the same account with
the key, while the server sees only authenticated ciphertext.

### Phase 4: hosted-edition feature enforcement

- Introduce a server-known deployment edition/capability policy.
- Disable Agents, workspace/shell/git tools, MCP stdio, managed backends, local
  plugins, native paths, and Electron-only controls.
- Ensure hidden renderer controls have matching route/service denial.
- Make direct provider transport the default hosted mode.
- Add a separately gated stateless relay only after SSRF and logging review.

Exit criterion: a hostile client cannot invoke disabled local capabilities or use
provider/MCP URLs to reach internal infrastructure.

### Phase 5: operations, billing, and documents

- Add subscription state without storing unnecessary customer identity in the
  encrypted workspace database.
- Separate workspace, billing, support, and analytics origins/services.
- Configure EU-region storage where chosen, encrypted backups, deletion windows,
  alerting, and incident response.
- Produce Impressum, privacy notice, terms, consumer cancellation/withdrawal
  information, provider-disclosure UI, and copyright/license inventory.
- Add a legal-notice channel and authority-request runbook.

Exit criterion: the rendered production site and actual vendor/data flow match the
documents exactly.

### Phase 6: security review and closed beta

- Threat-model database compromise, malicious account creation, XSS, dependency
  compromise, frontend replacement, auth-verifier theft, nonce reuse, rollback,
  sync conflicts, SSRF, relay logging, and account deletion failure.
- Run dependency, application-boundary, and infrastructure audits.
- Perform an external penetration test focused on authentication, tenant
  isolation, encrypted storage, and relay egress.
- Start with invite-only accounts, conservative quotas, no public sharing, and no
  adult marketing.

Exit criterion: critical/high findings are closed, legal launch checklist is
signed off, backups and deletion are tested, and public claims match verified
behavior.

## 12. Required security regression tests

- The server cannot derive the data key from stored account records.
- An authentication verifier copied from the database cannot be replayed as a
  login credential.
- Authentication and encryption derived keys differ and match published vectors.
- AES-GCM nonce reuse is structurally prevented.
- Modified ciphertext, AAD, wrapped DEK, or revision fails decryption.
- A stale server revision cannot silently replace a newer client revision.
- One account cannot enumerate, read, update, or delete another account's chunks.
- Logs, traces, metrics, error payloads, and support bundles contain no master key,
  provider key, prompt, response, or decrypted workspace value.
- Relay redirects and DNS rebinding cannot reach private/local/link-local/metadata
  addresses.
- Hosted clients cannot reach local Agents, MCP process, plugin install, shell,
  git, or filesystem routes.
- Account deletion removes live access immediately and expires backups according
  to the published retention period.
- The offline recovery utility can decrypt a valid export without contacting
  Vellium infrastructure.

## 14. Go/no-go rules

Do not launch the hosted edition if any of these remain true:

- the operator is unwilling to publish the legally required name/address;
- the service has no valid business address;
- the server can derive workspace decryption keys;
- provider credentials are stored using the current plaintext provider column;
- cross-account access is not covered by automated tests;
- arbitrary provider/MCP URLs can reach private infrastructure;
- hosted Agents, shell, filesystem, or local MCP execution remains reachable;
- the landing page markets adult content without a separately reviewed adult
  access model;
- public privacy claims exceed what the browser delivery and relay architecture
  actually prove;
- deletion, incident response, and backup expiration have not been tested.
